## 16) NVIDIA NIM API Rate-Limit Protocol (STRICT, NO-COMPROMISE, HIGH PRIORITY)

> **⚠️ CRITICAL PRIORITY:** This section is **STRICTLY ENFORCED on a NO-COMPROMISE basis**.
> Violating this protocol causes `Too Many Requests: {"status":429,"title":"Too Many Requests"}` errors which **disrupt workflow, waste tokens, and block progress**. This is the **HIGHEST PRIORITY operational constraint** for the agent.

### 16.1 The Problem

NVIDIA NIM API providers enforce a **hard limit of 30 requests per minute** across different models. When the agent sends multiple requests rapidly (burst pattern), the API provider returns:

```
Too Many Requests: {"status":429,"title":"Too Many Requests"}
```

This error **MUST be prevented proactively**, not reacted to after occurrence.

### 16.2 Mandatory Rate-Limit Rules (STRICT, NO-COMPROMISE)

| Rule | Requirement | Enforcement |
|------|-------------|-------------|
| **R1** | **NEVER send multiple requests simultaneously** | Agent MUST process ONE request at a time. NO parallel/burst requests to the same API provider. |
| **R2** | **Minimum 3-second gap between consecutive requests** | Agent MUST wait ≥3 seconds between any two consecutive API requests to the model provider. |
| **R3** | **Maximum 30 requests per rolling 60-second window** | Agent MUST track request count and self-throttle before hitting the limit. |
| **R4** | **Batch operations into SINGLE requests** | When multiple operations are needed, combine them into ONE request rather than sending many small requests. |
| **R5** | **No rapid retries on failure** | If a request fails, the agent MUST wait ≥5 seconds before retrying (NOT immediately retry). |
| **R6** | **Consolidate context** | The agent MUST curtail context length as needed to avoid unnecessary token consumption that triggers additional requests. |
| **R7** | **Exponential backoff on 429** | If a 429 error occurs despite precautions: wait 5s → 10s → 20s → 40s (exponential). Never retry more than 3 times. |
| **R8** | **No concurrent model calls** | When using different NVIDIA NIM models, the agent MUST serialize calls — NEVER call multiple models in parallel. |

### 16.3 Agent Self-Throttling Protocol

The agent **MUST** implement this internal protocol before EVERY API request:

```
BEFORE SENDING ANY REQUEST:
  1. Check: How many requests have I sent in the last 60 seconds?
     - If ≥25 → WAIT until the rolling window drops below 25 (leave 5-request safety margin)
  2. Check: How long since my last request?
     - If <3 seconds → WAIT until 3 seconds have elapsed
  3. Check: Is this request necessary, or can I batch it with pending work?
     - If batchable → DEFER and combine into a single later request
  4. Only after all checks pass → SEND the request (ONE at a time)
```

### 16.4 Context Curtailment Protocol (429 Prevention)

This is a **STRICT, NO-COMPROMISE** requirement.

#### 16.4.2 Context Curtailment Rules (STRICT, NO-COMPROMISE)

To minimize the number of requests needed (and thus avoid 429 errors), the agent **MUST**:

1. **Curtail context aggressively** — Remove redundant information, prior conversation history, and verbose examples when not needed. Target: **stay under 50% of model's max context** per request.
2. **Summarize instead of re-read** — If a file was already read, summarize from memory rather than re-reading it. NEVER re-read a file that was already read in the current session.
3. **Batch edits** — Use a SINGLE `replace_in_file` with multiple SEARCH/REPLACE blocks rather than multiple separate edit calls.
4. **Single comprehensive responses** — Deliver complete work in ONE response rather than fragmenting across multiple requests.
5. **Maintain healthy time gaps** — The agent MUST maintain a healthy time gap between requests to the model to prevent 429 errors. This is a **TOP PRIORITY** constraint.
6. **STRICT context limit per model** — Before sending ANY request, the agent MUST check: "Is my context size within the Practical Usage Limit for the active NVIDIA NIM model?" If not → **CURTAIL NOW** by summarizing, truncating, or deferring non-essential information.
7. **No unnecessary file reads** — The agent MUST NOT read files unnecessarily. If the information is already in context (from a previous read), use it. Do NOT re-read to "confirm" — this wastes context and triggers unnecessary requests.
8. **Compact conversation history** — When conversation history grows large, the agent MUST proactively summarize and compact it rather than letting it accumulate to hit context limits.
