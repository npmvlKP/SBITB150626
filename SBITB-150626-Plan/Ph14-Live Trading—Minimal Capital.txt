class SlippageCalibrator:
    """Measure real slippage and calibrate the paper trading model.

    Per Natenberg Ch.19: real fills always differ from theoretical prices.
    Per Kaufman Ch.12: slippage in live is always >= paper trading.
    Per Correction #10: calibrate after 30 real fills.

    Records for EVERY fill:
    - signal_price: when strategy decided to trade
    - submission_price: when kite.place_order() was called
    - fill_price: actual execution price from exchange
    - bid/ask at submission time

    Always positive (unfavorable to trader).
    """

        alert_dispatcher: AlertDispatcher,

        self._alert_dispatcher = alert_dispatcher
        self._fills: list[LiveFill] = []

        """Record a fill for slippage analysis."""

    def check_slippage_alert(self, fill: LiveFill) -> None:
        """Check if fill slippage exceeds SLIPPAGE_ALERT_MULTIPLIER.

        If real_slippage > 2x model_slippage: send WARNING alert.
        """

    def calibrate(self) -> SlippageCalibrationResult | None:
        """Calibrate SlippageModel parameters based on real fills.

        Called after REAL_SLIPPAGE_CALIBRATION_FILLS (30) fills.

        Steps:
        1. Compute avg_real_slippage_pct across all fills
        2. Compute avg_model_slippage_pct from fill records
        3. Compute calibration_ratio = avg_real / avg_model
        4. If ratio > 1.0: increase base slippage in model
        5. If ratio < 1.0: likely measurement error — log WARNING, don't decrease
        6. Return SlippageCalibrationResult with recommended new values
        7. Mark as calibrated
        """

    def get_slippage_stats(self) -> dict:
        """Get slippage statistics for monitoring."""

### 8.2 Key Implementation Rules:

- For buys: slippage = max(0, (fill_price - signal_price) / signal_price)
- For sells: slippage = max(0, (signal_price - fill_price) / signal_price)
- If real slippage is consistently < model, investigate: likely data issue, NOT better execution
- Calibration updates PaperTradingSettings BASE_SLIPPAGE values — propagated to future paper sessions
- After calibration, send INFO alert with old vs new slippage parameters
- Minimum 30 fills before calibration (statistical significance per Davey Ch.7)

### 12.2 Key Implementation Rules:
- CRITICAL: Never call kill_switch.deactivate() automatically after live crash

- On recovery failure (broker API unreachable): use last known state, send CRITICAL alert, PAUSE

STEP 23: EMERGENCY PROCEDURES
================================================================================

### Kill Switch Activation Paths (per MiFID II Art.17, Ph0 design):
1. **Telegram /kill** — most accessible, works from phone
2. **Keyboard Ctrl+Shift+K** — on server (Oracle Cloud SSH session)
3. **REST API POST /kill-switch** — programmatic
4. **Auto-kill triggers:**
   - Daily loss limit exceeded at 100%
   - Margin utilization ≥ 95%
   - Order rejection rate ≥ 10/minute
   - Position mismatch detected

### Kill Switch Response Protocol:
1. Kill switch activates → KILL level
2. All open orders cancelled (cancel_all_orders)
3. No new orders accepted
4. Telegram CRITICAL alert sent
5. Positions remain OPEN (do NOT auto-square-off — may cause worse fills)
6. Human decides: manually square-off via broker app, or /resume to let bot manage exits

### Crash Recovery Protocol:
1. Process crashes → systemd restarts (Ph13)
2. LiveCrashRecoveryManager.recover() executes
3. All open orders cancelled
4. Full position reconciliation
5. PAUSE activated (regardless of reconciliation result)
6. CRITICAL Telegram alert
7. Human must /resume after verifying everything is correct

### Broker API Failure Protocol:
1. If kite.place_order() fails with NetworkException: retry with exponential backoff (3 attempts)
2. If all retries fail: log CRITICAL, PAUSE, alert
3. If kite.get_positions() fails: use last known state, log WARNING
4. If kite.get_margins() fails: use last known margin (if < 5 min old), else PAUSE
5. If authentication fails (TokenException): attempt re-auth, if fails 3 times: PAUSE
