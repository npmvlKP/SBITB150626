INSTRUCTION 3.1 — Implement Gate: ruff lint
---------------------------------------------
Command:

  ruff check . --config pyproject.toml

Zero-Tolerance Criteria:
  - Exit code 0 = PASS
  - Exit code != 0 = FAIL (block merge)
  - NO --fix in CI; --fix allowed only in local dev
  - All errors must be resolved, not suppressed

Error Handling:

  ruff check . --config pyproject.toml; if ($LASTEXITCODE -ne 0) {
      ruff check . --config pyproject.toml --output-format=json > ruff-errors.json
      Write-Error "RUFF LINT FAILED. See ruff-errors.json for details."
      Write-Error "Run 'ruff check . --fix' locally to auto-fix safe issues."
      exit 1
  }

Verify:

INSTRUCTION 9.1 — Error Handling Matrix
------------------------------------------
Every gate failure follows this protocol:

  1. CAPTURE: Tool output saved to JSON/HTML artifact
  2. CLASSIFY: Severity assigned (BLOCK / WARN / INFO)
  3. COMMUNICATE: Error message with remediation guidance
  4. CONTAIN: Non-zero exit code prevents merge/release
  5. CORRECT: Developer fixes + re-runs gate locally before push
  6. CONFIRM: CI re-runs full gate; partial runs NOT allowed

Error Classification:

  | Tool          | BLOCK Condition              | WARN Condition           | INFO         |
  |---------------|------------------------------|--------------------------|--------------|
  | ruff check    | Any error                    | N/A                      | N/A          |
  | ruff format   | Any unformatted file         | N/A                      | N/A          |
  | mypy          | Any error                    | N/A                      | N/A          |
  | bandit        | HIGH severity                | MEDIUM severity          | LOW          |
  | pytest        | Any test failure             | Skipped tests            | N/A          |
  | coverage      | < 80% branch                 | 80-90%                   | > 90%        |
  | pip-audit     | Any vulnerability            | Deprecation warning      | N/A          |
  | gitleaks      | Any secret detected          | N/A                      | N/A          |
  | hypothesis    | Any falsifying example       | Flaky (fix with @seed)   | N/A          |
  | benchmark     | > 50% regression             | 20-50% regression        | < 20%        |
  | trivy         | HIGH/CRITICAL CVE            | MEDIUM CVE               | LOW CVE      |
  | cyclonedx     | Generation failure           | Missing NTIA field       | N/A          |
  | locust        | Error rate >= 1%             | P95 > 500ms              | P95 < 500ms  |
  | mutation      | Score < 60%                  | Score 60-80%             | Score > 80%  |

INSTRUCTION 9.3 — Security Exception Process
-----------------------------------------------
If a gate failure requires an exception:

  1. Document in SECURITY_EXCEPTIONS.md:
     - Tool, finding ID, severity, description
     - Risk assessment (likelihood x impact)
     - Compensating controls
     - Remediation timeline
     - Approver name + date

  2. Add to allowlist/ignore file:
     - gitleaks: .gitleaks.toml [[allowlist]]
     - bandit: # nosec comment with ID
     - trivy: .trivyignore with CVE ID + expiry
     - pip-audit: pip-audit --ignore-vuln ID (NOT recommended; fix instead)

  3. Review cadence: All exceptions reviewed quarterly

INSTRUCTION 9.4 — Non-Determinism Eradication Protocol
---------------------------------------------------------
Per Martin Fowler "Eradicating Non-Determinism in Tests":

  1. ISOLATE: Tests must not share mutable state
  2. FREEZE TIME: Use freezegun or clock wrapper for time-dependent tests
  3. MOCK EXTERNAL: All I/O (network, filesystem, database) via test doubles
  4. QUARANTINE: Flaky tests moved to quarantine suite immediately
  5. FIX: Quarantined tests fixed within 24 hours or removed
  6. SEED: Hypothesis tests use explicit @seed for reproducibility
  7. CLEANUP: Test fixtures clean up all resources (temp files, ports, processes)
